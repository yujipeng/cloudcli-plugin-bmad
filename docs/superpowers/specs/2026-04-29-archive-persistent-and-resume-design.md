# Persistent Archive And Resume-From-Version Design

## Context

当前插件的版本流转语义偏向“一次性归档”：当前工作区在达到某种完成条件后才出现归档建议，归档后形成只读历史版本，而后续推进默认只围绕当前工作区进行。这和当前目标不一致。

本次设计要解决两个明确需求：

1. 归档入口需要一直存在，不受“当前版本是否完成”影响。
2. 用户需要从某个归档版本恢复到当前工作区继续推进，并且在后续再次归档时，直接覆盖回原来的版本号，而不是生成一个新的版本号。

这意味着版本号在这里更接近“版本身份”，而不是“不可变快照编号”。当前工作区仍然是唯一可写区域；归档版本可以重新成为工作的起点，但恢复后再次归档会回写到原版本。

## Goal

让 BMad Flow 支持三种稳定行为：

- 当前工作区始终可归档
- 任一归档版本都可以恢复到当前工作区继续推进
- 从归档版本恢复出来的当前工作区再次归档时，覆盖回原版本

## Non-Goals

本次不实现以下能力：

- 归档版本的修订历史（例如 v1-r1 / v1-r2）
- 归档前自动备份旧版本
- 多个并行当前工作区
- 从同一归档版本派生多个分支工作区
- 历史版本差异对比视图
- 高级恢复模式（例如恢复后解除归档目标绑定）

## User Model

版本系统采用以下语义：

- 当前工作区：唯一可写区
- v1 / v2 / v3：版本身份
- 从归档版本恢复到当前工作区后，当前工作区会带上“归档目标版本”标记
- 之后再次归档时，直接覆盖原目标版本目录

这套语义下：

- 普通当前工作区首次归档 -> 生成新的版本号
- 从 v1 恢复出来的当前工作区再次归档 -> 覆盖写回 v1

代价是 v1 不再是不可变快照；收益是版本号稳定，符合“继续推进同一个版本”的直觉。

## UX Design

### 1. 当前版本页中的归档卡片常驻

当前版本页始终显示归档卡片，不再要求当前版本“已完成”才出现。

状态分两种：

#### 普通当前工作区
- 标题：归档当前版本
- 说明：归档为 vN
- 按钮：归档为 vN

#### 从归档版本恢复出来的当前工作区
- 标题：归档当前版本
- 说明：当前工作区来自 v1，归档时将覆盖 v1
- 按钮：归档回 v1

#### 禁用状态
仅当当前工作区完全为空时禁用归档。

- 标题：归档当前版本
- 说明：当前工作区为空，无可归档内容
- 按钮：禁用

### 2. 归档版本页中的继续推进操作

在每个归档版本页增加一个新操作：

- 按钮：继续为当前版本
- 说明：复制到当前工作区继续推进，后续归档将回写到该版本

这个按钮只在 `version.kind === 'archived'` 的版本页里显示。

### 3. 冲突提示

如果用户想从归档版本恢复，但当前工作区里已经有内容，则不允许覆盖恢复。

提示文案：
- 请先归档当前工作区，再从历史版本继续推进

这条规则比“直接覆盖当前工作区”更安全，也更符合现有版本心智模型。

## Backend Semantics

### 1. 归档模式

当前工作区需要带一个由后端计算并返回的归档模式：

- `new`：创建新版本号
- `overwrite`：覆盖回原版本
- `disabled`：当前工作区为空，禁止归档

同时在覆盖模式下返回目标版本号：

- `archiveTargetVersionId: 'v1'`

### 2. 从归档版本恢复到当前工作区

新增一个恢复动作，例如：

- `POST /version/continue-as-current`

请求体：

```json
{
  "path": "/project/path",
  "sourceVersionId": "v1"
}
```

服务端流程：

1. 校验项目路径
2. 校验 `sourceVersionId` 对应的归档版本存在
3. 校验当前工作区是否为空
4. 若不为空，返回 409 与冲突原因
5. 若为空，则把 `docs/v1/planning-artifacts/*` 复制到当前工作区的 `planning-artifacts`
6. 把 `docs/v1/implementation-artifacts/*` 复制到当前工作区的 `implementation-artifacts`
7. 写入当前工作区来源标记，使后续归档模式变成 `overwrite`
8. 返回最新的 `VersionedResponse`

### 3. 当前工作区来源标记

需要一个轻量持久化标记，记录当前工作区是从哪个归档版本恢复出来的。

建议存成项目内的小元数据文件，而不是只保存在内存里。原因是：

- 刷新插件后仍要保持“归档回 v1”的行为
- 服务端重启后状态不能丢
- `/versions` 接口需要稳定复算归档模式

建议存储内容：

```json
{
  "archiveMode": "overwrite",
  "archiveTargetVersionId": "v1"
}
```

当当前工作区是普通模式时，这个文件不存在，或被清空为默认状态。

### 4. 再次归档时的处理

`POST /version/archive-current` 在进入归档时，不再只判断“建议归档到下一个版本号”，而是要分两种路径：

#### 普通归档
- 目标版本号来自当前已有归档版本最大号 + 1
- 行为与现有归档逻辑基本一致：把当前工作区移动到新目录，再重建空的当前工作区

#### 覆盖归档
- 目标版本号来自 `archiveTargetVersionId`
- 行为改为覆盖写回该版本目录
- 完成后重建空的当前工作区
- 清除当前工作区的来源标记

## File Responsibilities

### `src/index.ts`
负责版本页 UI、归档卡片渲染、版本切换、按钮交互。

本次需要：
- 让归档卡片常驻
- 支持显示 `archiveMode` 与 `archiveTargetVersionId`
- 在归档版本页显示“继续为当前版本”按钮
- 调用新的 `POST /version/continue-as-current`
- 处理“当前工作区非空”的冲突提示

### `src/types.ts`
负责前后端共享的版本流转类型。

本次需要扩展：
- 归档模式类型
- 当前工作区归档目标版本字段
- 继续推进操作的允许状态与原因

### `src/versionDiscovery.ts`
继续只负责发现当前工作区和版本目录，不承担归档策略决策。

本次不应把“覆盖回原版本”的业务规则塞进这里，避免它变成状态机文件。

### `src/archiveExecutor.ts`
负责实际文件系统归档动作。

本次需要扩展为两种执行模式：
- 新建归档
- 覆盖归档

其中覆盖归档要支持“目标目录已存在”的情况，并把当前工作区内容覆盖写回目标版本目录。

### `src/server.ts`
负责版本数据组装、归档接口、恢复接口。

本次需要：
- 在 `/versions` 的 `VersionedResponse` 中加入当前工作区归档模式信息
- 改造 `POST /version/archive-current`，根据元信息决定新建归档或覆盖归档
- 新增 `POST /version/continue-as-current`
- 统一校验当前工作区是否为空
- 在恢复与归档后都返回新的版本视图数据

### `tests/server/versionDiscovery.test.ts`
负责版本发现与流转行为测试。

本次至少补齐四类测试：
- 当前工作区未完成时仍然给出归档能力
- 从归档版本恢复到当前后，返回 `archiveMode=overwrite` 与目标版本号
- 从归档版本恢复后再次归档，覆盖回原版本，不生成新版本号
- 当前工作区非空时，恢复归档版本返回冲突

## Minimal Implementation Scope

为了降低风险，这次实现只做一条完整闭环：

1. 归档卡片常驻
2. 归档版本可恢复到当前工作区
3. 恢复后再次归档覆盖回原版本
4. 当前工作区非空时禁止恢复

不做额外历史、分支、备份、diff 能力。

## Error Handling

### 恢复失败
- 找不到源版本 -> 404
- 当前工作区非空 -> 409
- 文件复制失败 -> 500

### 归档失败
- 普通归档目标目录已存在 -> 500 / 409
- 覆盖归档时目标目录不存在 -> 404 或 500
- 文件移动/复制失败 -> 500

错误消息要直接返回给前端，用于 alert 或后续更友好的 UI 提示。

## Testing Strategy

### Server tests
补齐服务端测试覆盖：

- 常驻归档逻辑
- 恢复到当前工作区
- 恢复后的覆盖归档
- 恢复冲突

### UI regression checks
最小 UI 验证至少覆盖：

- 当前页能看到常驻归档卡片
- 归档版本页能看到“继续为当前版本”
- 恢复后当前页按钮从“归档为 vN”切换成“归档回 v1”

## Trade-offs

### 优点
- 归档入口更稳定，用户不必等“完成态”才管理版本
- 用户可以围绕同一个版本号持续迭代
- 当前工作区仍是唯一可写区，模型足够简单

### 缺点
- 归档版本不再是不可变快照
- 覆盖归档意味着历史内容会被新内容替换
- 如果以后要保留版本修订历史，需要额外设计 revision 层

## Recommended Rollout

建议按下面顺序落地：

1. 先补类型与 `/versions` 返回字段
2. 再做恢复接口与当前工作区来源标记
3. 然后扩展归档接口支持覆盖模式
4. 最后改前端文案和按钮状态
5. 用测试锁住恢复/覆盖归档语义

## Final Recommendation

采用以下最终方案：

- 归档入口始终存在
- 当前工作区为空时禁用归档
- 从归档版本恢复时，复制到当前工作区
- 恢复后当前工作区带来源版本标记
- 再次归档时，直接覆盖原版本目录
- 当前工作区非空时，必须先归档当前工作区，才能从归档版本继续推进

这套方案与当前诉求完全对齐，而且是在现有代码结构上最小、最稳定的一条实现路径。
